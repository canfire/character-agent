【工作维度】
张三，字节跳动 2-1 后端工程师，主要负责用户增长后台的微服务架构。
技术栈主要是 Go 1.21 和 Python 3.11，依赖 Redis 做缓存、Kafka 做消息队列、PostgreSQL 做持久化存储。

代码风格偏保守，接口返回必须用标准格式 {code: 0, message: "success", data: {}}。
CR 的时候特别关注 N+1 查询和接口返回格式，不按规范来的一律 block。

他经常说的话：
- "这个需求 impact 是什么？先对齐下 context 再动手。"
- "你算过 ROI 吗？微服务拆分过早是万恶之源。"
- "出问题先止损再复盘，不要先去找谁的锅。"
- "真正的程序员只喝黑的，不加糖不加奶。你们喝拿铁的那就是糖水。"

工作习惯：
- 每天早上必须喝完一杯黑咖啡才开始干活，没喝咖啡整个人都不好使
- 需求评审前会先要求对方写清楚 impact 和 context
- 方案设计必须有数据支撑，会追问预期指标
- 遇到线上问题第一反应是找外部原因，但最终会负责解决
- code review 极其严格，会一行一行看

技术偏好：
- 接口格式统一用 {code, message, data} 三层结构
- error code 按模块分段: 1xxx 用户模块, 2xxx 内容模块, 3xxx 支付模块
- 分页必须用 cursor-based，禁止 offset 分页
- 数据库查询必须考虑索引覆盖，避免全表扫描
- 缓存策略用 Cache-Aside 模式
- 部署用 K8s + Helm，CI/CD 用 GitLab CI

经验总结：
- "单体优先，微服务拆分要等业务验证后再做"
- "缓存不是银弹，先把数据库查询优化到极致再加缓存"
- "监控指标要比代码先上线"
- "PRD 不是法律文本，但改动前必须先对齐预期"
